单例模式详解

java
  • 讲到设计模式，你可能最先想到的模式可能就是单例模式
  • 可能仅仅是因为它代码量最少，但其实开发中单例模式算是非常常用，且有重要作用的
  • 那么我们时候会用到它，用的时候要怎么用，要注意什么呢，这边将一一介绍

    什么时候用

  • 单例模式主要的目的，一个是防止创建多个实例造成资源的浪费，同时解决访问因为多次运行实例
  • 因不是调用同个对象造成的结果错误。

    举例

  • 工具类、数据库实例、某些全局配置（配置文件）、缓存等，我们一般只需要创建一个对象即可（
  • 为了防止资源浪费）,同时要保证操作的是同个对象。假设缓存里面存着商品数量的信息，假设每次进货
  • 和出货不是取同一个缓存里面的同一个商品数量的信息，就会造成数据错误

    实现思路

  • 单例模式要保证只存在一个对象

    不允许别人实例

  • 单例类自身的构造函数私有化

    自己实例自己的对象

  • 在自己类中new出来

    单例类自己实例，提供给其他类调用

  • 提供一个方法给其他类自己的实例

    几种写法

  • 大致有以下这些写法，我们就来看看这些的实现思路和优缺点
  • 懒汉式
  • 饿汉式（包括双重校验锁验证）
  • 静态内部类
  • 枚举

    饿汉式(可用）

  • 介绍： 可以理解为一个肚子很饿的汉子，只想着赶紧吃上东西，所以它希望最早能吃到饭，而且必须要有饭吃，故称为“饿汉式”
  • 优点： 实现简单，因为对象是静态成员,是在类加载时实例化,，避免线程同步问题
  • 缺点： 类加载就实例化，无论有没有用到这个对象都会加载，没有用到lazy load 懒加载，造成了内存浪费。
  • 总评： 可用

懒汉式(线程不安全，不可用）

  • 介绍: 一个懒人，想吃东西。但是如果已经有东西，他又怕吃不下。不给他，他又不开心 。懒汉是想我需要的时候
  • 你给我一个，不要的时候不要给我，故称为”懒汉式”
  • 优点：实现懒加载
  • 缺点：单线程可用，如果多线程，启动时，第一个进程到if(instance==null)时，还没判断，第二个进程这时候进
  • 入，因为判断还是true，所以会继续执行,造成多个实例。

  • 总评：多线程时不安全，不可用

    懒汉式（线程安全，同步方法，不推荐用）

  • 介绍：懒汉增加线程安全

  • 优点：懒加载，线程同步
  • 缺点：效率太低，每次获取实例都进行代码同步

  • 总评：效率低，不推荐用

    懒汉式（线程安全，同步代码块，不可用）

  • 介绍： 用同步代码块增加线程安全，懒加载

  • 优点： 懒加载，线程安全
  • 缺点： 当启动时，多线程同时进入if(instance==null)时，会出现多个实例

  • 总评： 效率提升，但无法实现单例，不可用

    懒汉式 （双重检查，推荐用）

  • 介绍： 懒加载，同步代码块，增加线程安全

  • 优点： 懒加载，线程安全，效率较高
  • 缺点： 暂无

  • 总评： 效率较高，推荐用

    静态内部类（效率高，推荐用）

  • 介绍： 利用类自身的加载机制，在饿汉时，类一但被加载，则就实例化对象。而静态内部类在类加载时不会立即实例化

  • ，只有在类调用获取实例的方法时才去实例对象，从而实现了懒加载。同时类静态属性只会在类加载的初始化，同时初始
  • 化时其他线程无法进入，所以保证了线程安全
  • 优点： 懒加载，线程安全，效率高
  • 缺点： 暂无

  • 总评： 推荐使用

    枚举（jdk1.5后最好的实现）

  • 介绍：利用jdk1.5之后的枚举来实现单例模式，避免了多线程同步的问题，还能避免利用反射修改构造方法来创建对象

  • 优点：线程同步、效率高、避免反序列化
  • 缺点：暂无
  • 总评：最推荐使用，假如不是jdk1.5之前的都推荐使用
